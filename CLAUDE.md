# Bonsai App 開発ガイド

盆栽管理アプリ（Bonsai App）の開発における指示とルールをまとめたファイルです。

## アプリケーション概要

盆栽の管理を支援するWebアプリケーション。主な機能：

- **ユーザー認証**: ID/パスワードによるログイン（AWS Cognito）
- **盆栽管理**: 登録、編集、一覧表示、詳細表示
- **作業記録**: 剪定、植替え、水やり、肥料などの作業記録の管理
- **作業予定**: 今後の作業予定の管理
- **画像管理**: 盆栽や作業の画像アップロード・表示

## データモデル

- **User**: ユーザー情報（ID、メール、名前など）
- **Bonsai**: 盆栽情報（名前、樹種、登録日、来歴、画像URLなど）
- **WorkRecord**: 作業記録（盆栽ID、作業タイプ、日付、説明、画像URLなど）
- **WorkSchedule**: 作業予定（盆栽ID、作業タイプ、予定日、説明、完了フラグなど）

## 技術スタック

- **Node.js**: v16.x以上
- **Angular**: v13.x（クライアント）
- **AWS**: Lambda / S3 / DynamoDB / Cognito / CloudFormation
- **AWS CLI**: 最新版

## 開発フロー

### ブランチ戦略

- `main`: 本番環境用（pushで本番デプロイ）
- `develop`: 開発環境用（pushで開発環境デプロイ）
- 機能開発は `feature/機能名` ブランチで行う

### 各工程の定義

**1. 要件定義・設計工程**
- ユーザーの要求をインタビューして分析し、システム要件を洗い出す
- 要件承認後に `feature/` ブランチを作成し、設計ドキュメントを `docs/design/` に追記・修正してコミット
- 設計完了後、`docs/test/` に結合テストケースを作成する
- 結合テストは設計をもとに因子と水準を洗い出し、ペアワイズ法で2因子間を網羅する組み合わせテストケースとして作成

**2. 製造工程**
- `feature/` ブランチにコミット済みの設計ドキュメントをもとに TDD でコードを実装
- Red → Green → Refactor のサイクルを守る
- ユニットテストを実行して確認後、`feature/` ブランチを `develop` にマージ

**3. 結合テスト工程**
- テスト環境情報: `docs/test/.env`
- ログイン後、テストケースに従ってテストを実行
- 想定結果と実際の結果を比較し、テスト結果レポートを作成
- レポート承認が得られたら開発完了

### TDD の原則

1. **Red**: 失敗するテストを先に書く
2. **Green**: テストが通る最小限のコードを実装する
3. **Refactor**: コードをリファクタリングする

### コミットメッセージ形式

```
[コンポーネント] 変更内容
```

例: `[server] ユーザー認証機能の実装`

## AWS 無料枠の制限

コストを意識し、以下の制限内に収まるよう設計・実装する：

- **Lambda**: 月100万リクエスト、400,000 GB秒
- **S3**: 5GBストレージ、20,000 GET/月、2,000 PUT/月
  - フロントエンド用と画像保存用の2バケットに分割
  - 画像は適切に圧縮してアップロード（2MB以上は段階的圧縮）
  - 推奨形式: WebP（最効率）、対応形式: JPG / PNG / GIF（アニメーションGIF非推奨）
- **DynamoDB**: 25GBストレージ、読み書きキャパシティを最小設定
- **Cognito**: 50,000 MAU

## トラブルシューティング

| 問題 | 対処法 |
|------|--------|
| npm install エラー | `node_modules` と `package-lock.json` を削除して再試行 |
| ビルドエラー | `dist` を削除して再試行、TypeScript バージョン互換性を確認 |
| デプロイエラー | AWS CLI 認証情報と CloudFormation テンプレート構文を確認 |
| 画像アップロードエラー | S3 バケットの CORS 設定を確認 |
